# Building for Web (WASM)

This document describes a practical first pass for running Schism Tracker in a
browser with Emscripten.

## Current status of this port scaffold

The `scripts/build-web.sh` script added in this repository is designed to get a
minimal browser build pipeline in place quickly.

- Uses `configure` + `make` through `emconfigure` / `emmake`
- Forces SDL2 backend
- Disables host-only backends (ALSA/JACK/X11)
- Automatically works around source paths containing spaces
- Produces browser artifacts in `build-web/dist`

This is an incremental starting point, not a complete production web port.

For Emscripten builds, this scaffold currently uses a minimal internal UTF-8
mapping fallback in place of external `utf8proc` linkage to keep first-pass WASM
builds moving.

## Prerequisites

- Emscripten SDK (`emcc`, `emconfigure`, `emmake`) available in `PATH`
- Usual Schism build tools:
  - `autoconf`, `automake`, `libtool`, `pkg-config`, `python3`, `perl`

On macOS/Linux, after installing emsdk:

```sh
source /path/to/emsdk/emsdk_env.sh
```

## Build

From repository root:

```sh
./scripts/build-web.sh
```

Artifacts will be in:

- `build-web/dist/schismtracker.js`
- `build-web/dist/schismtracker.wasm`
- `build-web/dist/schismtracker.data` (if generated by linker flags)
- `build-web/dist/index.html`

## Run locally

Any static web server works. For example:

```sh
cd build-web/dist
python3 -m http.server 8080
```

Open [http://localhost:8080](http://localhost:8080).

## Known web-specific constraints

- Audio usually starts only after user interaction.
- Browser shortcuts can conflict with tracker keybindings.
- Native filesystem paths are unavailable; use import/export + persistent web
storage patterns.
- Threading is intentionally disabled in this scaffold to avoid COOP/COEP setup
requirements in the first iteration.
- Web MIDI depends on browser support and user permission; some browsers or contexts (for example non-secure origins) may not allow it.

### Configurable app shortcuts

The web build includes a shortcut configuration page:

- Open via `System Configuration (Ctrl-F1) -> Shortcuts: Configure...`
- Rebind button captures a new key chord for the selected action
- Duplicate bindings are rejected and the conflicting action is shown
- Defaults / Save / Back are available as on-screen buttons

In some browser/OS combinations, `Ctrl+F1` may be intercepted before it reaches
the app. If this happens, use `Ctrl+Shift+F1` to open shortcut configuration.

### IndexedDB persistence (IDBFS)

The generated `index.html` mounts Emscripten’s IDBFS at `/persistent`, sets
`HOME=/persistent`, and delays `main()` until `syncfs(true)` completes (via
`addRunDependency` / `removeRunDependency` so startup and storage stay in sync).
Settings under `~/.config/schism` and default module/sample directories live on
that path; opened modules are copied into `/persistent/modules/` for reuse.
`syncfs(false)` runs on unload, tab hide, and periodically. If IDBFS is
unavailable (e.g. some private modes), the open flow falls back to `/tmp` only.

The web toolbar also provides stored module management for `/persistent/modules`
through a `Stored Action` dropdown plus `Run Action` button:

- `Load Stored`: load currently selected stored module
- `Refresh List`: rescan IndexedDB-backed storage
- `Rename Stored`: rename the selected stored module (sanitized filename)
- `Delete Stored`: remove the selected stored module

The stored list is sorted by most recently modified file first.

Display scaling is selectable from the toolbar:

- `Fit Window`: stretch to available viewport area (default)
- `x1`: fixed `640x400` logical size
- `x2`: fixed `1280x800`
- `x3`: fixed `1920x1200`

### Web MIDI (input/output)

The web runner initializes the browser Web MIDI API and exposes one virtual
`Web MIDI I/O` port in Schism's MIDI page.

- Incoming MIDI messages from browser MIDI inputs are forwarded to Schism's MIDI
  event pipeline.
- Outgoing MIDI messages generated by Schism are sent to all currently available
  browser MIDI outputs.
- Browser permission and secure-context requirements apply (`localhost` is OK).
